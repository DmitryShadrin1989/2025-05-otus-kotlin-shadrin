# User Story для проекта «Электронная записная книжка»

## 1. Разработчик (студент‑автор проекта)

**User Story 1: Реализация базового функционала записей**  
Как разработчик, я хочу реализовать CRUD‑операции для записей, чтобы пользователи могли управлять своими заметками.

**Критерии приёмки:**
- Пользователь может создать новую запись с заголовком и текстом.
- Пользователь может просмотреть список своих записей.
- Пользователь может открыть и прочитать конкретную запись.
- Пользователь может отредактировать существующую запись.
- Пользователь может удалить запись.
- Все операции сохраняются в базе данных.

**User Story 2: Реализация аутентификации**  
Как разработчик, я хочу внедрить систему аутентификации, чтобы обеспечить безопасность пользовательских данных.

**Критерии приёмки:**
- Пользователь может зарегистрироваться, указав email и пароль.
- Система проверяет корректность ввода данных при регистрации.
- Пользователь может войти в систему, используя свои учётные данные.
- При неверном пароле система отображает ошибку.
- После входа пользователь получает токен доступа.
- Токен используется для авторизации последующих запросов.

## 2. Преподаватель (куратор проекта)

**User Story 1: Проверка реализации требований**  
Как преподаватель, я хочу иметь доступ к демо‑версии приложения, чтобы оценить соответствие проекта техническому заданию.


**Критерии приёмки:**
- Приложение развёрнуто и доступно для тестирования.
- Все заявленные функции работают согласно спецификации.
- Код соответствует принятым стандартам оформления.
- Документация проекта полная и понятная.
- Разработчик может объяснить архитектурные решения.

## 3. Пользователи (конечные потребители)

**User Story 1: Создание и хранение записей**  
Как пользователь, я хочу создавать и хранить личные записи, чтобы не забывать важные дела и мысли.

**Критерии приёмки:**
- Я могу быстро создать новую запись в один клик.
- Запись сохраняется автоматически или по нажатию кнопки.
- Мои записи доступны при следующем входе в систему.
- Я вижу список всех своих записей с заголовками.
- Записи отображаются в хронологическом порядке (новые сверху).

**User Story 2: Поиск записей**  
Как пользователь, я хочу искать свои записи по ключевым словам, чтобы быстро находить нужную информацию.

**Критерии приёмки:**
- В интерфейсе есть поле поиска.
- При вводе текста система фильтрует записи в реальном времени.
- Поиск работает по заголовку и содержимому записей.
- Если совпадений нет, отображается сообщение «Записей не найдено».

**User Story 3: Организация записей по категориям**  
Как пользователь, я хочу группировать записи по категориям (например, «Работа», «Дом», «Покупки»), чтобы легче ориентироваться в заметках.

**Критерии приёмки:**
- Я могу создать новую категорию.
- При создании записи я могу выбрать категорию из списка.
- Я могу фильтровать записи по выбранной категории.
- В списке записей отображается метка категории.
- Я могу редактировать или удалять категории.

**User Story 4: Напоминания и уведомления**  
Как пользователь, я хочу устанавливать напоминания для записей, чтобы не пропустить важные события.

**Критерии приёмки:**
- При создании/редактировании записи я могу установить дату и время напоминания.
- Система отправляет уведомление за 15 минут до указанного времени.
- Я могу отключить напоминание для конкретной записи.
- В списке записей отображается значок, если установлено напоминание.

## 4. Тестировщики (одногруппники/коллеги)

**User Story 1: Проведение функционального тестирования**  
Как тестировщик, я хочу иметь чек‑лист для проверки всех функций приложения.

**Критерии приёмки:**
- Существует документ с тестовыми сценариями для каждой функции.
- Для каждого сценария указаны шаги выполнения и ожидаемый результат.
- Результаты тестирования фиксируются в отчёте.
- Отчёт включает найденные дефекты с приоритетом и описанием.

## 5. Технические консультанты (наставники/эксперты)

**User Story 1: Анализ архитектурных решений**  
Как технический консультант, я хочу изучить архитектуру приложения, чтобы дать рекомендации по оптимизации.

**Критерии приёмки:**
- Представлена схема архитектуры системы (компоненты, связи).
- Описаны используемые технологии и их обоснование.
- Есть описание механизмов безопасности.
- Консультант может предложить улучшения по производительности или масштабируемости.

## 6. Системный администратор

**User Story 1: Развёртывание приложения**  
Как системный администратор, я хочу развернуть приложение на сервере, чтобы оно было доступно пользователям.


**Критерии приёмки:**
- Приложение успешно устанавливается на целевой сервер.
- Все зависимости (БД, сервер приложений) настроены корректно.
- Приложение доступно по указанному URL.
- Логи системы записываются и доступны для мониторинга.
- Настроены резервные копии данных.

## 7. Дизайнер интерфейса

**User Story 1: Разработка пользовательского интерфейса**  
Как дизайнер, я хочу создать удобный и понятный интерфейс для работы с записями.

**Критерии приёмки:**
- Интерфейс содержит все необходимые элементы (список записей, форма создания, поиск).
- Дизайн соответствует принципам UX/UI (интуитивность, доступность).
- Элементы управления имеют понятные подписи и иконки.
- Интерфейс адаптивен под разные размеры экрана.
- Цветовая схема не напрягает глаза и соответствует тематике приложения.

## 8. Аналитик требований

**User Story 1: Сбор и анализ требований**  
Как аналитик, я хочу собрать и формализовать требования к системе, чтобы разработчики понимали, что нужно реализовать.

**Критерии приёмки:**
- Составлен документ с функциональными и нефункциональными требованиями.
- Требования структурированы по приоритетам.
- Каждое требование имеет чёткое описание и критерии проверки.
- Документ согласован с заинтересованными сторонами.

## 9. Владелец продукта

**User Story 1: Управление бэклогом задач**  
Как владелец продукта, я хочу вести список задач и приоритизировать их, чтобы эффективно планировать разработку.

**Критерии приёмки:**
- Существует актуальный бэклог с перечнем всех функций и улучшений.
- Каждая задача имеет оценку сложности и приоритет.
- Бэклог регулярно обновляется на основе обратной связи.
- Есть план релизов с указанием сроков реализации функций.
- Владелец продукта может обосновать приоритеты задач.
